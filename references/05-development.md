# 阶段五：开发跟进 — 做开发的加速器，不是减速带

> PM 在开发期的价值不是「催进度」，而是「清障碍」：模糊需求、频繁变更、没人管的外部依赖。

## 每日节奏

- **站会（15 min）**：PM 是观察者，关注「阻塞项」非「进度%」。有人被阻塞 → 记下会后立刻跟进。
- **异步同步（推荐）**：扫 `#daily-sync`，发现「阻塞」字段非空立即跟进。

**保护开发的三个动作**
1. 做需求防火墙：任何人中途提需求，PM 先接住评估，不让需求直飞开发。
2. 统一信息入口：所有澄清走 PM，不让开发收矛盾需求。
3. 保护大块时间：编码高峰期不排长同步会。

## 里程碑检查

| 节点 | 检查 |
|------|------|
| 30% | 核心流程能跑通？UI 基本可见？ |
| 70% | 功能基本完整？关键体验 OK？ |

30% 掉头成本低；70% 是大规模纠偏最后机会，再往后是上线前赶工。

## 风险预警

`实际进度落后计划 >20% → 立即升级`：① 与 TL 确认偏差原因；② 评估对上线日影响；③ 备两方案（A 砍范围保时间 / B 延时间保范围）；④ 带方案找 Stakeholder 决策，不只报问题。

## 需求变更管理

收到变更先答三问：① 不变更的代价（线上 Bug 影响 50% 用户 vs 交互更好看）？② 真实成本（开发+测试+回归+文档）？③ 值不值得现在做（能否下个迭代）？

| 类型 | 处理 |
|------|------|
| 线上 Bug >10% 用户 | 立刻 Hotfix |
| 合规/安全风险 | 立刻修，不计成本 |
| 需求理解偏差致 P0 不满足 | 必须改 |
| 体验微调 | 记优化清单，本迭代不做 |
| 「我觉得更好」 | 「记下了，下迭代优先评估」 |

## PM 在开发期易犯的错误

| 错误 | 正确做法 |
|------|----------|
| 频繁问「做完了吗」 | 看进度看板 |
| 中途加新需求 | 走变更或排下迭代 |
| 绕过 TL 找开发 | 通过 TL 协调 |
| 联调才看实现 | 30%/70% Demo |
| 测试才写验收 | PRD 阶段写完 |
| 技术之争站开发对立面 | 一起找折中 |

## 产出：进度/风险周报

`本周交付（✅/🟡/🔴）| 下周计划 | 阻塞项 | 技术决策 | 风险提醒`。同时触发阶段六衔接提示。
