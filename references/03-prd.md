# 阶段三：PRD 撰写 — 写好一份让工程师想实现的文档

> PRD 的目标不是「写得完整」，而是「写得清楚」。完整但没人看得懂 = 没写。

## 黄金标准

1. 没参与讨论的人看完能理解 80%，不需当面解释。
2. 工程师看完知道每个功能点的验收标准。
3. 设计师看完知道要解决的核心问题（不是「照竞品做」）。

## PRD 十章（逐章要点）

**① 问题陈述**：三段式（当前状态 / 影响范围 / 机会大小）。用数据锚定，如「35% 搜索零结果，每周 120 工单」。
**② 目标（Goals）**：每个 SMART。如「零结果率 35%→<15%（上线 2 周）」。
**③ 非目标（Non-Goals）**：最易跳过、最防范围蔓延。每条必须解释「为什么不做」。
**④ 用户故事**：`作为[具体角色]，我希望[行为]，以便[价值]`。角色用行为特征（「5 人以下团队负责人」非「用户」）。分 P0/P1/P2/P3。
**⑤ 功能需求**：每条 = 描述 + 验收标准 + 优先级。验收标准用 **Given-When-Then**。
**⑥ 交互/设计参考**：给设计师上下文（设计系统复用、竞品参考、约束），不是替设计师画线框。
**⑦ 成功指标**：领先指标（上线当天可观测：零结果率、响应 P99、搜索使用率）+ 滞后指标（1-4 周：打开率、工单量、留存）。每个指标 **基线 + 目标值**。
**⑧ 风险与依赖**：技术/依赖/体验/合规，每项有应对方案。
**⑨ 待决策问题**：问题 + 决策人 + 截止 + 影响范围。
**⑩ 上线计划**：灰度阶梯 + 自动停止条件 + 监控告警。

## Given-When-Then 示例

```
REQ-01 模糊搜索匹配（P0）
Given 文档标题「会议纪要-7月3日」
When 用户搜索「7月会议」
Then 该文档出现在相关结果前 3 位
Given 用户输入错别字「会义」
When 系统模糊匹配
Then 匹配「会议纪要」并标注「您是不是要找：会议」
Given 搜索结果为零
When 无匹配
Then 展示引导文案 + 「创建新文档」入口
```

## 异常状态必须覆盖（质量试金石）

| 场景 | 需描述 |
|------|--------|
| 空状态 | 无数据/首次使用 |
| 加载中 | 慢网/大数据过渡 |
| 错误 | 网络/权限/服务异常 |
| 边界 | 极大极小数据、特殊字符 |
| 权限 | 不同角色内容差异 |

## PRD 质量自检清单（门禁，详见 assets/review-checklist.md）

□ 问题陈述：陌生人 30 秒看懂？ □ 目标可衡量？ □ Non-Goals ≥3 条且每条有原因？
□ 用户故事角色-行为-价值齐全？ □ 每个 P0 有 Given-When-Then？ □ 五类异常状态覆盖？
□ 指标有基线+目标值？ □ 隐私合规已标注？ □ Tech Lead 已确认可行性？ □ 外部依赖已标注且对方知情？
