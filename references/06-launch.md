# 阶段六：上线验收与迭代 — 上线不是终点

> 上线的完成不是「功能发布了」，而是「指标达到了预期」。没有数据佐证的「感觉不错」= 自我安慰。

## 1. 验收测试（PM 亲自按 PRD 逐条走）

**功能**：P0 用户故事逐条过；所有 Given-When-Then 通过；权限控制正确。
**异常**：空状态、断网/超时、输入异常（空/超长/特殊字符）、快速重复操作、后台切换数据不丢。
**体验一致**：文案无错、符合设计系统、过渡/错误态视觉一致。
**性能兼容**：核心操作 P99 达标、目标设备/浏览器通过、弱网基本可用。
**数据埋点**：关键事件上报正确、参数完整（用户 ID/时间戳/上下文）、能答「谁何时做了什么」。
**隐私合规**：告知同意、存储传输合规、用户可查看/删除数据。

## 2. 灰度发布（最小代价验证最大假设）

```
内部员工(1天) → 1%(1天) → 10%(1-2天) → 50%(1天) → 100%
每阶段检查：功能完整 / 稳定性(错误率·Crash·P99) / 核心指标(A/B) / 客服反馈
```

**自动停止条件（必须设定）**
1. 错误率 > 基线 2 倍
2. Crash 率 > 0.5%
3. 核心业务指标统计显著负向（p<0.05）
4. 单日 >N 条与新功能直接相关的投诉

## 3. 上线后数据复盘

| 时间 | 行动 |
|------|------|
| 当天 | 监控看板无异常告警 |
| 第 3 天 | 领先指标是否在预期内 |
| 第 7 天 | 初步复盘（核心指标/反馈/Bug） |
| 第 14 天 | 完整复盘（滞后指标 + 是否达目标） |

**复盘模板**
```
【功能】上线复盘  日期：
1. 指标达成：| 指标 | 基线 | 目标 | 实际 | 达成？ |
2. 用户反馈摘录（正/负）
3. 意外发现
4. 遗留问题（Bug 编号/优先级）
5. 下一步
```

## 4. 功能下线决策（会做减法才是高级 PM）

满足 ≥2 项启动下线评估：使用率 <5% 活跃且 2 月无增长 / 维护成本高（Bug 占比>15%）/ 战略偏离 / 反效果（降核心指标）。

**下线流程**：数据分析(1天) → 决策审批(1-2天, PM+总监) → 用户沟通(提前≥1周: 公告+替代方案+迁移) → 迁移期(2-4周, 允许导出/一键迁移) → 正式关闭(删代码+更新文档)。

⚠️ 反面案例：某 SaaS 未通知下线导出功能，200+ 企业投诉、30+ 流失。**不告而取是大忌**。

## 5. 闭环意识

`需求→分析→PRD→评审→开发→上线→复盘→发现新需求→↑闭环`。复盘必问：假设验证了吗（对错）？重来哪能更好？下一个相关需求是什么？
