# 阶段二：需求分析与优先级排序 — 学会说「不」

> 做产品的本质不是「做什么」，而是「不做什么」。每一次「不」都是对用户时间的尊重。

## 1. 需求分析五问（深度拆解）

**第一问：为谁解决什么问题？**
「为谁」要具体可识别的行为特征（非人口属性）；「问题」描述当前状态用户的感受或行为受阻。
差：「解决协作体验不好」。好：「为首次用协作功能的小团队管理者，解决不知如何开始第一次协作的困惑」。

**第二问：不解决会怎样？（量化影响）**
按可信度：① 直接数据（工单量/投诉数）② 行为推断（流失率 × 用户数）③ 用户原话（≥5 人主动提同一问题）④ 类比估算。影响 <1% 且无增长影响 → 砍。

**第三问：现在是最好的时机吗？**
战略契合度 / 市场窗口（竞品 3 月内会做？客户合同 deadline？）/ 资源状态 / 技术基础。两个维度答「否」则时机不对。

**第四问：有没有更简单的方式？**
能用文案解决的不用交互，能用交互解决的不用开发，能用现有功能组合的不用新功能。

**第五问：我们来做有什么优势？**
竞品花同样资源也能做且做得更好 = 追平（要做但优先级低于差异化）。

## 2. RICE 框架（实操）

`RICE = (Reach × Impact × Confidence) / Effort`

**Reach（覆盖，单周期影响用户数）**：10=80%+ / 7-8=50-80% / 4-6=20-50% / 1-3=5-20% / 0.5=<5%。

**Impact（对北极星指标提升）**：3=>10% / 2=5-10% / 1=1-5% / 0.5=<1% / 0.25=极微小。

**Confidence（把握）**：100%=A/B 验证 / 80%=访谈+数据 / 50%=类比推断 / 20%=纯直觉。

**Effort（人月，实事求是）**：估时不准是所有延期的根源，不要为推需求压低估时。

**局限性（何时不用 RICE）**：战略性需求（老板拍板）、合规/安全（必须做）、全新领域（无数据支撑 Confidence）。

## 3. 辅助框架

**MoSCoW**（时间紧必须砍）：Must ≤60% / Should ~20% / Could ~15% / **Won't ≥5%（必须非空，否则说明不敢说不）**。

**Kano**（功能设计阶段）：基本型（不做用户愤怒，做了理所应当，做到 80 分即可）/ 期望型（越好越满意）/ 兴奋型（不做没事，做了惊喜，把精力放这）。

## 4. 需求优先级决策会

**会前**：所有需求入表预填 RICE 初始分；标记 ≤3 个争议项；提前发表不现场念。
**会中（45 min）**：0-5 回顾上次执行；5-15 过非争议项；15-35 讨论争议（每项 ≤7 min）；35-45 确认排序+匹配容量+输出迭代计划。
**纪律**：讨论超 10 min 无结论 → 标记离线跟进；排序依据是数据逻辑不是职级；PM 是会议主人防止被主导。
