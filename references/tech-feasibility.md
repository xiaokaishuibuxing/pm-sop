# 技术可行性评估 — 防「画饼」

> PM 不写代码，但必须能判断方案是否在工程现实内。画出不切实际的饼，伤的是团队信任。

## 1. 四维快速评估（与 Tech Lead 对齐前 PM 自评）

| 维度 | 自问 | 红旗信号 |
|------|------|----------|
| 复杂度 | 核心难点在哪？业界有无成熟方案？ | 「用 AI 一键搞定」「应该不难吧」 |
| 依赖 | 需哪些外部系统/团队？对方排期确定吗？ | 依赖方未知或已排满 |
| 性能 | 数据量级？延迟/并发要求？ | 要实时但数据源 T+1 |
| 风险 | 最坏情况是什么？有降级方案吗？ | 无降级、单点故障无备份 |

## 2. 与 Tech Lead 对齐的提问清单

1. 「这个方案技术上可行吗？有没有更经济的实现？」
2. 「最大技术风险点是什么？需要提前做技术 spike 吗？」
3. 「依赖的 XX 团队排期你确认过吗？」
4. 「预估多少人天？乐观/悲观区间？」
5. 「有没有我们低估的隐性成本（迁移/回归/运维）？」
6. 「如果砍掉 20% 范围，哪部分能省掉且不影响核心价值？」

## 3. 「画饼」预警信号（PM 自查）

- 方案描述全是形容词（「智能」「无缝」「一键」），无具体交互/数据流向。
- 宣称解决一个业界未解难题（如「完全准确的语义搜索」）但无技术路径。
- Effort 估时显著低于同类历史需求。
- 依赖尚未立项的外部能力（「等中台做好了就能用」）。
- 用「AI/大模型」规避对准确率、延迟、成本的量化承诺。

## 4. 与 PRD 衔接

技术预评审结论必须回写 PRD：在第⑧章更新技术方案/风险/依赖，更新 Effort 评估（阶段二 RICE）。未过技术预评审的 PRD 不得进正式评审（见 `references/04-review.md`）。
